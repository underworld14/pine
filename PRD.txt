Menurutku kalau **build from scratch**, jangan jadikan targetnya "alternatif Beads". Itu akan membuat scope-mu terlalu sempit.

Dari semua obrolan kita, aku melihat visinya lebih tepat seperti ini:

> **"The local-first project workspace for AI-assisted development."**

Issue tracker hanyalah salah satu fiturnya.

# Product Requirements Document (PRD)

## Project Name (Working Title)

**Pine** 🌲 *(working title)*

> Git-native local workspace for AI-assisted software development.

---

# Vision

Menjadi workspace lokal yang membantu developer mengelola hasil vibecoding, QA, bug, feature request, dan konteks proyek, sehingga manusia dan AI dapat bekerja pada sumber data yang sama.

Semua data disimpan sebagai file di repository.

Tidak ada cloud.

Tidak ada akun.

Tidak ada database.

---

## Ada satu perubahan yang menurutku sangat penting

Aku **tidak akan menjual produk ini sebagai "issue tracker"**.

Aku akan memosisikannya sebagai:

> **"AI Development Workspace"**

Karena issue hanyalah satu bagian. Yang benar-benar membedakan adalah workflow lengkap:

**Build → Test → Report → Fix with AI → Verify**, semuanya berbasis file lokal di dalam repository.

Dengan positioning itu, ruang pengembangan produk akan jauh lebih luas daripada sekadar bersaing dengan Beads, Linear, atau Jira. Kamu bisa berkembang menjadi "tool pendamping" untuk Claude Code, Codex, Gemini CLI, atau agen AI lain, bukan hanya aplikasi untuk mencatat tiket.
